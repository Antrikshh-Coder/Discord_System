📋 Written Analysis: What Fails First?

1
Memory explodes first — All messages appended to a single in-memory list. At 4,200 msg/sec × 150 bytes = ~37 MB/min. After 1 hour: 2.2 GB. No eviction, no pagination.
2
CPU collapses under fan-out — Broadcasting one message to 50,000 subscribers in a single thread locks the event loop. New messages queue up. Latency spikes from ms to seconds.
3
The hotspot is the killer — 80% of traffic hits one channel. Even if you had 100 servers, routing ALL of channel #cricket-live to one node defeats the purpose.
4
Network bandwidth saturates — Broadcasted WebSocket frames to 50K clients: if each frame is 200 bytes and you send 4,200/sec, that's 840 MB/sec — impossible on a standard NIC.

Introducing Shards — But No Strategy Yet

You now have 3 machines. Where does data go? The routing logic is intentionally missing — that's the problem.

Python shards.py — No Routing Logic 
class Shard:
    def __init__(self, shard_id):
        self.id = shard_id
        self.messages = []  # Each shard is INDEPENDENT ✓

    def store(self, message):
        self.messages.append(message)

class ShardManager:
    def __init__(self, num_shards):
        self.shards = [Shard(i) for i in range(num_shards)]

    def send_message(self, message):
        # ❓ Intentionally incomplete — where does it go?
        pass

🤔 Key Insight: Routing is Everything

Having shards means nothing without a routing strategy. Randomly assigning breaks queries. Routing everything to shard 0 defeats the purpose. The right answer depends on your access patterns — and this assignment explores why every naive answer is flawed.

Day 8 — Better But Not Perfect

Hash-Based Sharding — The Right Direction

MD5 hash of a key gives uniform distribution. But which key you hash changes everything. And what happens when you add more shards?

Python hash_shard_manager.py 
import hashlib

class HashShardManager(ShardManager):

    def get_shard(self, key):
        h = int(hashlib.md5(str(key).encode()).hexdigest(), 16)
        return self.shards[h % len(self.shards)]

    def send_message(self, message):
        # 🤔 Student Decision: What do you hash?
        # hash(user_id) → same as user sharding
        # hash(channel_id) → uniform channels, but hotspots remain
        # hash(message_id) → best distribution, worst for queries
        key = message.channel_id  # ← chosen here
        shard = self.get_shard(key)
        shard.store(message)

🔑 Choose Your Hash Key:

This is the critical decision. Each choice has different consequences.

⚠️ Resharding Problem: What Happens When Shards Change?

3 shards → key % 3
User 42 → shard 0 (42%3=0)
User 43 → shard 1 (43%3=1)
User 44 → shard 2 (44%3=2)
⟶ Add a shard ⟶
4 shards → key % 4
User 42 → shard 2 (42%4=2) ← MOVED!
User 43 → shard 3 (43%4=3) ← MOVED!
User 44 → shard 0 (44%4=0) ← MOVED!

🚨 ALL existing data is now on the WRONG shard. Every query fails until data is migrated.

🔍 Cross-Shard Query: Fetch Last 10 Messages from #cricket-live

After distributing messages across shards, how do you retrieve a channel's history?

Python cross_shard_query.py 
def fetch_channel_messages(manager, channel_id, limit=10):
    results = []
    shards_checked = 0

    for shard in manager.shards:
        shards_checked += 1
        # Filter messages for this channel from each shard
        channel_msgs = [
            m for m in shard.messages
            if m.channel_id == channel_id
        ]
        results.extend(channel_msgs)

    # Sort by timestamp, take last N
    results.sort(key=lambda m: m.timestamp)
    print(f"Checked {shards_checked} shards for {len(results)} total messages")
    return results[-limit:]

Day 10 — Final Analysis

Deep Analysis — Understanding the Trade-offs

Which shard failed first? What looks good but collapses under a spike? What breaks when shards scale from 3 → 10?

Q1

Which shard failed first and why?

In User-Based sharding: Shard 0 fails first because the top influencer (user_id=0) is always routed to it. With 5,000+ messages from a single user vs ~200 from others, Shard 0 carries 5–10× the load of Shard 1 or 2.

In Channel-Based sharding: Shard 1 fails first during the cricket event because channel_id=1 maps to Shard 1. One channel with 80% traffic overwhelms it while Shard 0 and Shard 2 sit nearly idle.

Shard overloaded → write latency spikes
↓
Message queue backs up → in-memory buffer grows
↓
OOM kill → shard goes down → data loss for that partition
Q2

Which strategy looked good but failed under spike?

Hash(channel_id) looks balanced on a normal day — channels spread evenly across shards. But during a cricket event, the ONE hot channel still all goes to one shard. The hash just changes which shard gets obliterated.

The false sense of security is the real danger: your monitoring shows "even distribution" on 49 channels, while the 50th channel kills a shard.

hash("cricket-live") % 3 = 2
4,200 msg/sec → Shard 2 → 💥
Q3

What happens when shards increase from 3 → 10?

Naive hashing breaks completely: With mod-3, User 42 → Shard 0. With mod-10, User 42 → Shard 2. Every single existing key maps to a different shard. You cannot read any existing data correctly.

The fix: Consistent hashing (like a hash ring) minimizes remapping — only 1/N keys move when a new shard is added. Discord uses this in production.

Q4

What breaks when one shard goes down?

Immediate effects: All writes to that shard fail. Messages in that partition are inaccessible. Users on channels homed to that shard see errors.

Cascading effects: Other shards receive redirected traffic → they start overloading too → cascade failure possible.

Data implications: Without replication, all messages on the dead shard are LOST. You must either replicate each shard (2× cost) or accept data loss.

Data Loss
100% of that shard
Service Degradation
~33% of users affected
Recovery Time
Minutes to hours

📈 System Evolution: 3 → 6 Shards

When you double shards from 3 to 6 using key % num_shards:

  • Every key remaps to a new shard (100% data migration)
  • During migration, queries return wrong data
  • The system is effectively unavailable during migration
  • For 1 billion messages, this could take DAYS

Consistent hashing places both shards and keys on a virtual ring. Adding a shard only moves ~1/N of keys:

  • 3 → 6 shards: only ~50% of keys move (half to new shards)
  • 3 → 4 shards: only ~25% of keys move
  • Migration can happen gradually, live, without downtime

Discord's actual approach (as of their engineering blogs):

  • Uses Cassandra with vnodes (virtual nodes) for consistent hashing
  • Each "shard" maps a range of guild IDs (Discord's term for servers)
  • Hot guilds (large servers) get dedicated shards
  • Added "Super Sharding" for guilds with >250,000 members
  • Read replicas for popular channels during events
